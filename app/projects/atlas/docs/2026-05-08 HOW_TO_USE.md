# Atlas — 产业链知识图谱引擎 使用指南

> 本文档面向 **第一次接触 Atlas** 的开发者和运维人员，详细介绍如何搭建、配置和使用 Atlas。
>
> Atlas 是一个基于 LLM 的产业链知识图谱引擎，能够从财报、研报、新闻等文本中自动抽取结构化知识（公司、行业、产品、资源、事件等），构建 Neo4j 知识图谱，并提供影响分析、产业链追踪等能力。

---

## 一、系统架构概览

```
┌──────────────────────────────────────────────────────────────────┐
│                          Atlas (Python)                          │
│                   Industry Chain KG Engine                        │
│                                                                  │
│  ┌──────────┐  ┌─────────────┐  ┌───────────┐  ┌──────────────┐ │
│  │ Document  │  │  LLM        │  │  Graph    │  │  Impact      │ │
│  │ Ingestion │→│  Extraction  │→│  Builder  │→│  Analysis    │ │
│  └──────────┘  └─────────────┘  └───────────┘  └──────────────┘ │
│       ↑                                ↓                         │
│    MinIO                         PhoenixA Client                 │
└──────────────────────────────────┬───────────────────────────────┘
                                   │ HTTP
                                   ↓
┌──────────────────────────────────────────────────────────────────┐
│                       PhoenixA (Go)                              │
│                  Data Middleware Layer                            │
│                                                                  │
│  ┌──────────────┐  ┌──────────────┐  ┌────────────────────────┐ │
│  │ KG Domain    │  │ Graph API    │  │ Security Domain        │ │
│  │ (PostgreSQL) │  │ (Neo4j)      │  │ (PostgreSQL+Timescale) │ │
│  └──────────────┘  └──────────────┘  └────────────────────────┘ │
└──────────────────────────────────────────────────────────────────┘
```

**关键设计决策**：
- Atlas **不直接连接** Neo4j 或 PostgreSQL，所有数据操作通过 PhoenixA HTTP API 完成
- PhoenixA 是统一的数据中间件，管理所有数据库连接和生命周期
- Atlas 专注于 **智能层**：LLM 抽取、知识图谱构建、影响分析

## 二、依赖服务清单

Atlas 运行需要以下服务就绪：

| 服务 | 用途 | 是否必须 | 默认地址 |
|------|------|----------|----------|
| **PhoenixA** | 数据中间件（PostgreSQL + Neo4j 代理） | ✅ 必须 | `http://127.0.0.1:18085` |
| **Neo4j** | 知识图谱存储 | ✅ 必须 | `bolt://localhost:7687` |
| **PostgreSQL** | KG 元数据（文档、抽取记录、事件等） | ✅ 必须 | PhoenixA 管理 |
| **LLM API** | 文本抽取（DeepSeek / OpenAI 兼容） | ✅ 必须 | `https://api.deepseek.com` |
| **MinIO** | 文档文件存储 | ❌ 可选 | `localhost:9000` |
| **RSS 源** | 每日新闻自动抓取 | ❌ 可选 | 配置中定义 |

## 三、环境准备

### 3.1 安装 Neo4j

**Docker 方式（推荐）**：

```bash
docker run -d \
  --name neo4j \
  -p 7474:7474 -p 7687:7687 \
  -e NEO4J_AUTH=neo4j/your_password \
  -e NEO4J_PLUGINS='["apoc"]' \
  -v neo4j_data:/data \
  neo4j:5-community
```

安装完成后：
- Web 控制台：`http://localhost:7474`
- Bolt 协议：`bolt://localhost:7687`
- 默认用户/密码：`neo4j / your_password`

### 3.2 安装 MinIO（可选）

```bash
docker run -d \
  --name minio \
  -p 9000:9000 -p 9001:9001 \
  -e MINIO_ROOT_USER=minioadmin \
  -e MINIO_ROOT_PASSWORD=minioadmin \
  -v minio_data:/data \
  minio/minio server /data --console-address ":9001"
```

### 3.3 确保 PhoenixA 已配置 Neo4j + KG 数据源

PhoenixA 的 `config.yaml` 中需要启用：

```yaml
# PhoenixA config.yaml
neo4j:
  enabled: true
  uri: "bolt://localhost:7687"
  username: neo4j
  password: "your_password"
  database: neo4j
  max_connection_pool_size: 50

postgres_gorm:
  enabled: true
  data_sources:
    kg:                              # Atlas 用的 KG schema
      host: 192.168.1.100
      port: 5432
      user: chaos_app
      password: "your_pg_password"
      database: chaos_db
      schema: kg
      migrate_enabled: true
      migrate_base: ./migrations
```

然后启动 PhoenixA：

```bash
cd app/projects/phoenixA
go run cmd/main.go -config config/config.yaml
```

### 3.4 获取 LLM API Key

Atlas 使用 [litellm](https://github.com/BerriAI/litellm) 作为 LLM 客户端，支持多种模型提供商：

- **DeepSeek**（推荐）：注册 https://platform.deepseek.com/ 获取 API Key
- **OpenAI**：使用 OpenAI 格式的模型名，如 `gpt-4o`
- **任何 OpenAI 兼容 API**：配置 `api_base` 即可

设置环境变量：

```bash
export LLM_API_KEY="sk-your-api-key"
```

### 3.5 安装 Atlas Python 依赖

```bash
cd app/projects/atlas

# 创建虚拟环境（推荐）
python -m venv .venv
source .venv/bin/activate  # Linux/Mac
# .venv\Scripts\Activate.ps1  # Windows PowerShell

# 安装依赖
pip install -e ".[dev]"
```

## 四、配置文件详解

配置文件位于 `config/atlas.yaml`，完整结构如下：

### 4.1 `server` — 服务监听

```yaml
server:
  host: 0.0.0.0
  port: 18400
```

### 4.2 `neo4j` — Neo4j 连接（供 Atlas 内部感知，实际连接由 PhoenixA 管理）

```yaml
neo4j:
  uri: bolt://localhost:7687
  username: neo4j
  password: atlas_password
  database: neo4j
  max_connection_pool_size: 50
```

> ⚠️ 此配置仅用于 Atlas 内部参考。Atlas 实际通过 PhoenixA API 访问 Neo4j，请确保 PhoenixA 的 Neo4j 配置与此一致。

### 4.3 `skills` — 抽取技能 Prompt 配置（🔑 核心配置）

```yaml
skills:
  # 抽取技能 prompt 文件路径
  # 支持相对路径（相对于 atlas 项目根目录）或绝对路径
  extraction_prompt: "../../tools/py/skills/[2026-04-28]INDUSTRY_EXTRATION_SKILLS.md"
```

**说明**：
- 抽取技能 Prompt 是 Atlas 的核心资产，它决定了 LLM 如何理解和抽取结构化知识
- 当前使用的是 `[2026-04-28]INDUSTRY_EXTRATION_SKILLS.md`（v5 版本）
- 支持热切换：修改此路径后重启 Atlas 即可使用新版本 Prompt
- Prompt 文件可以用 Markdown 格式编写，内容会作为 LLM 的 system prompt
- 如需创建新版本，建议保留旧版本文件，用日期前缀命名（如 `[2026-05-08]xxx.md`）

**自定义 Prompt 要点**：
- 文件中必须定义 JSON 输出结构（nodes + edges）
- nodes 至少应包含 `companies`、`industries`、`products` 等字段
- edges 的 `type` 字段应与 `atlas.yaml` 中 `taxonomy.relationship_types` 对应

### 4.4 `llm` — LLM 模型配置

```yaml
llm:
  extraction_model: "deepseek/deepseek-v4-flash"    # 抽取用模型（快+便宜）
  filter_model: "deepseek/deepseek-v4-flash"         # 新闻过滤用模型
  summary_model: "deepseek/deepseek-v4-flash"        # 摘要用模型
  reasoning_model: "deepseek/deepseek-v4-pro"        # 推理/分析用模型（强）
  api_key: "${LLM_API_KEY}"                          # 环境变量引用
  api_base: "https://api.deepseek.com"               # API 基础 URL
  max_retries: 3                                      # 失败重试次数
  request_timeout: 120                                # 单次请求超时（秒）
  batch_concurrency: 3                                # 批量抽取并发数
```

**双模型策略**：
- `extraction_model` / `filter_model`：使用轻量快速模型，控制成本
- `reasoning_model`：影响分析、公司评估等复杂任务使用强模型

### 4.5 `minio` — 文档存储

```yaml
minio:
  endpoint: "localhost:9000"
  access_key: "${MINIO_ACCESS_KEY:minioadmin}"       # 支持默认值语法
  secret_key: "${MINIO_SECRET_KEY:minioadmin}"
  secure: false
  bucket: atlas-documents
```

> MinIO 不可用时 Atlas 仍可运行，但文档上传和重新解析功能会受限。

### 4.6 `document` — 文档处理参数

```yaml
document:
  storage_dir: ./data/documents       # 本地文档缓存目录
  chunk_max_chars: 3000               # 分块最大字符数
  chunk_overlap_chars: 200            # 分块重叠字符数
```

### 4.7 `pipeline` — 每日管道配置

```yaml
pipeline:
  news_sources:
    - type: rss
      name: sina_finance
      url: "https://finance.sina.com.cn/rss/"
    - type: rss
      name: eastmoney
      url: ""
  daily_max_news: 200                 # 每日最大抓取新闻数
  daily_max_extraction: 50            # 每日最大抽取文档数
```

### 4.8 `graph` — 图谱构建参数

```yaml
graph:
  company_merge_threshold: 0.85       # 公司名相似度合并阈值
  impact_max_hops: 3                  # 影响传导最大跳数
  low_confidence_threshold: 0.5       # 低于此置信度的节点/边不写入图谱
```

### 4.9 `dept_services` — 依赖服务地址

```yaml
dept_services:
  cronjob:
    host: "127.0.0.1"
    port: 19999
  phoenixA:
    host: "127.0.0.1"
    port: 18085
```

## 六、核心使用场景

### 6.1 场景一：手动上传文档 → 抽取 → 构建图谱

**步骤 1：上传文档**

```bash
curl -X POST http://localhost:18400/api/v1/documents/upload \
  -F "file=@宁德时代2025年报.pdf" \
  -F "doc_type=earnings" \
  -F "company_name=宁德时代"
```

响应：
```json
{
  "doc_id": "a1b2c3d4e5f67890",
  "title": "宁德时代2025年报.pdf",
  "chunk_count": 15,
  "status": "uploaded"
}
```

**步骤 2：触发抽取**

```bash
curl -X POST http://localhost:18400/api/v1/documents/a1b2c3d4e5f67890/extract
```

响应：
```json
{
  "doc_id": "a1b2c3d4e5f67890",
  "chunks_processed": 14,
  "nodes_created": 42,
  "edges_created": 38,
  "total_cost": 0.0123
}
```

这一步会自动：
1. 从 MinIO 下载原始文件
2. 文本提取（PDF → 纯文本）
3. 分块（按段落，3000 字/块，200 字重叠）
4. 对每个块调用 LLM 提取结构化数据
5. 通过质量门控（过滤低置信度结果）
6. 通过 PhoenixA → Neo4j 合并节点和边

### 6.2 场景二：浏览器书签栏一键提交

```bash
curl -X POST http://localhost:18400/api/v1/ingest/web-clip \
  -H "Content-Type: application/json" \
  -d '{
    "title": "锂价创两年新高",
    "url": "https://example.com/news/123",
    "content": "2026年5月，碳酸锂价格突破每吨15万元大关...",
    "doc_type": "news"
  }'
```

### 6.3 场景三：直接提交文本块抽取

```bash
curl -X POST http://localhost:18400/api/v1/extractions/trigger \
  -H "Content-Type: application/json" \
  -d '{
    "doc_id": "manual_test_001",
    "chunk_texts": [
      "宁德时代2025年营收达到4500亿元，同比增长35%。主要受益于储能业务的爆发式增长..."
    ]
  }'
```

### 6.4 场景四：每日自动管道

```bash
# 自动抓取 RSS → 过滤 → 抽取 → 构建图谱 → 事件去重 → 影响分析
curl -X POST http://localhost:18400/api/v1/pipeline/daily
```

也可以由 cronjob 服务定时触发，或手动传入新闻：

```bash
curl -X POST http://localhost:18400/api/v1/pipeline/daily \
  -H "Content-Type: application/json" \
  -d '{
    "news_items": [
      {"title": "锂价暴跌", "content": "碳酸锂价格跌破8万元/吨..."}
    ]
  }'
```

## 十、数据流转全流程

```
原始文本（PDF / HTML / 纯文本 / Web Clip / RSS 新闻）
    │
    ▼
[1] 文本解析 + 分块 (ingestion.py)
    │  PyMuPDF(PDF) / BeautifulSoup(HTML) / UTF-8(文本)
    │  按段落分块：3000 字/块，200 字重叠
    │
    ▼
[2] LLM 结构化抽取 (llm_extractor.py)
    │  System Prompt = skills.extraction_prompt 指定的文件
    │  输入：文本块
    │  输出：JSON（nodes + edges）
    │  模型：extraction_model (deepseek-v4-flash)
    │
    ▼
[3] Pydantic 校验 (models/graph_schema.py)
    │  验证 JSON 结构合法性
    │  过滤无效字段
    │
    ▼
[4] 质量门控 + 实体标准化 (graph_builder.py)
    │  过滤 confidence < low_confidence_threshold 的节点/边
    │  标准化公司名（normalized_name）
    │
    ▼
[5] 写入 Neo4j (via PhoenixA)
    │  MERGE 节点（按 label + merge_key 去重）
    │  MERGE 边（保留 attrs）
    │  批量写入（merge-batch API）
    │
    ▼
[6] 事件去重 + 影响分析（每日管道额外步骤）
    │  对抽取出的 Event 节点做指纹去重
    │  对新事件触发影响传导分析
    │
    ▼
[7] 存储抽取记录到 PostgreSQL (via PhoenixA)
       记录 doc_id, chunk_index, cost, tokens, status
```

## 十一、抽取技能 Prompt 版本管理

### 11.1 当前版本

当前使用 `[2026-04-28]INDUSTRY_EXTRATION_SKILLS.md`（v5），支持抽取以下实体和关系：

**实体类型**：Company, Industry, Market, Product, Technology, Asset, Resource, Policy, Event

**关系类型**：
- 产业链：SUPPLIER_OF, CUSTOMER_OF, PRODUCES, PART_OF_PRODUCT
- 资源：DEPENDS_ON_RESOURCE, CONSUMES_RESOURCE, PRODUCES_RESOURCE, EXTRACTS_RESOURCE
- 组织：SUBSIDIARY_OF, INVESTED_IN, COMPETITOR_OF
- 行业：BELONGS_TO_INDUSTRY, OPERATES_IN_MARKET
- 事件/政策：INVOLVED_IN_EVENT, IMPACTED_BY_POLICY, IMPACT_ON
- 技术/资产：USES_TECHNOLOGY, OWNS_ASSET, APPLIED_IN

### 11.2 切换 Prompt 版本

1. 将新 Prompt 文件放到 `app/tools/py/skills/` 目录（推荐）或任意位置
2. 修改 `config/atlas.yaml` 中 `skills.extraction_prompt` 路径
3. 重启 Atlas

```yaml
skills:
  extraction_prompt: "../../tools/py/skills/[2026-05-15]NEW_EXTRACTION_SKILLS.md"
```

### 11.3 编写新 Prompt 的注意事项

- 输出格式必须是合法 JSON，包含 `meta`、`nodes`、`edges` 三个顶级字段
- `nodes` 中各数组字段名（`companies`, `industries`, `products` 等）需与 `graph_builder.py` 中 `_LABEL_MAP` 对应
- `edges` 中 `type` 字段需与 `_REL_TYPE_MAP` 对应（或使用大写形式）
- 每个 node 和 edge 应有 `confidence` 字段（0~1），用于质量门控

## 十三、与其他服务的协作

### 13.1 Atlas ↔ PhoenixA

Atlas 通过 `phoenixa_client.py` 调用以下 PhoenixA API：

| `POST /api/v1/kg/impact-logs` | 记录影响分析 |
| `POST /api/v1/graph/cypher` | 执行只读 Cypher |
| `POST /api/v1/graph/cypher/write` | 执行写入 Cypher |
| `POST /api/v1/graph/nodes/merge` | 合并单个节点 |
| `POST /api/v1/graph/nodes/merge-batch` | 批量合并节点 |
| `POST /api/v1/graph/edges/merge` | 合并单条边 |

### 13.2 Atlas ↔ Cthulhu (前端)

Cthulhu 前端可通过 Atlas API 展示：
- 知识图谱可视化（`/api/v1/graph/*`）
- 影响分析报告（`/api/v1/analysis/*`）
- 文档管理（`/api/v1/documents/*`）

### 13.3 Atlas ↔ Cronjob

Cronjob 服务定时调用 Atlas 的 `/api/v1/pipeline/daily` 触发每日管道。

## 十五、常见问题

### Q1: Atlas 启动报错 "Cannot find extraction skill prompt"
**A**: 检查 `config/atlas.yaml` 中 `skills.extraction_prompt` 路径是否正确。路径相对于 Atlas 项目根目录（`app/projects/atlas/`）。

### Q2: 抽取不出任何实体？
**A**: 检查 LLM API Key 是否有效、模型名是否正确。可以先用 `/api/v1/extractions/trigger` 접口测试一小段文本。查看 Atlas 日志中的 LLM 调用记录。

### Q3: PhoenixA 连接失败？
**A**: 确认 `dept_services.phoenixA` 配置中的 host:port 与 PhoenixA 实际监听地址一致。用 `curl http://host:port/health` 验证。

### Q4: 图谱中节点重复？
**A**: 调整 `graph.company_merge_threshold` 参数。Graph Builder 使用 `normalized_name` 作为 Company 的去重键。确保 Prompt 中要求 LLM 输出 `normalized_name`。

### Q5: 每日管道没有抓到新闻？
**A**: 检查 `pipeline.news_sources` 中 RSS URL 是否配置且可访问。需要安装 `feedparser` 包。

### Q6: MinIO 不可用，Atlas 还能正常工作吗？
**A**: 可以。MinIO 故障时，文档上传仍会记录元数据，但文件不会持久化到对象存储。后续通过 `/{doc_id}/extract` 重新处理时需要文件可用。

### Q7: 如何估算 LLM 成本？
**A**: 每次抽取的成本记录在抽取结果的 `cost_usd` 字段中。每日管道结束后也会汇总 `total_cost_usd`。以 DeepSeek V4 Flash 为例，约 ¥0.01/千 tokens，一篇 3000 字的文档约 ¥0.03。
