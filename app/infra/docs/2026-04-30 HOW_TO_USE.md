# Go Infra 框架使用指南

> 本文档面向 **第一次接触此框架** 的开发者，详细介绍如何使用 Go Infra 框架搭建服务。
>
> Go Infra 是一个组件化的应用框架，通过 YAML 配置声明式地启用各种基础设施组件（HTTP 服务器、数据库、gRPC、日志、监控等），框架自动完成组件的初始化、生命周期管理和优雅关闭。

---

## 一、快速开始

### 1.1 最小可运行示例

```go
package main

import (
    "log"
    "github.com/grand-thief-cash/chaos/app/infra/go/application"
)

func main() {
    app := application.GetApp()
    if err := app.Run(); err != nil {
        log.Fatal(err)
    }
}
```

### 1.2 最小配置文件 (`config.yaml`)

```yaml
app_info:
  app_name: "my-service"
  env: "development"

logging:
  enabled: true
  level: "INFO"
  format: "console"
  output: "stdout"
```

### 1.3 启动命令

```bash
go run main.go -config config.yaml -env development
```

**命令行参数**：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `-config` | `config.yaml` | 配置文件路径（支持相对/绝对路径） |
| `-env` | `development` | 环境标识（`development` / `production`） |

### 1.4 框架启动流程

```
1. 解析命令行参数 (-config, -env)
2. 加载 YAML 配置文件
3. 根据配置中 enabled: true 的组件，自动创建并注册
4. 按依赖顺序启动所有组件
5. 阻塞等待 OS 信号 (SIGINT / SIGTERM)
6. 收到信号后，按逆序优雅关闭所有组件
```

## 二、配置文件结构全景

```yaml
# ─── 基础信息 ───────────────────────────────────────────
app_info:
  app_name: "my-service"           # 应用名（日志/监控/telemetry 中使用）
  env: "development"               # 环境

# ─── 日志 ──────────────────────────────────────────────
logging:
  enabled: true                    # 是否启用（false = 无日志输出）
  # ... 详见下文

# ─── HTTP 服务器 ────────────────────────────────────────
http_server:
  enabled: true
  # ...

# ─── HTTP 客户端 ────────────────────────────────────────
http_clients:
  enabled: true
  # ...

# ─── gRPC 服务器 ────────────────────────────────────────
grpc_server:
  enabled: false
  # ...

# ─── gRPC 客户端 ────────────────────────────────────────
grpc_clients:
  enabled: false
  # ...

# ─── MySQL (GORM) ──────────────────────────────────────
mysql_gorm:
  enabled: false
  # ...

# ─── PostgreSQL (GORM) ─────────────────────────────────
postgres_gorm:
  enabled: false
  # ...

# ─── Redis ──────────────────────────────────────────────
redis:
  enabled: false
  # ...

# ─── Neo4j ──────────────────────────────────────────────
neo4j:
  enabled: false
  # ...

# ─── Prometheus 监控 ──────────────────────────────────
prometheus:
  enabled: false
  # ...

# ─── Telemetry (OpenTelemetry) ─────────────────────────
telemetry:
  enabled: false
  # ...

# ─── 业务配置 ──────────────────────────────────────────
biz_config:
  # 任意自定义结构，框架不解析，由业务代码读取
```

> **核心规则**：每个组件都有 `enabled` 字段。`enabled: false` 或整个 section 不写 → 组件不会被创建和启动。

## 三、各组件详细配置参考

### 3.1 `app_info` — 应用基础信息

```yaml
app_info:
  app_name: "phoenixA"             # 应用名称（必填）
  env: "development"               # 环境: development | production
```

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `app_name` | string | ✅ | 应用名。用于日志文件名、telemetry service name 等 |
| `env` | string | ❌ | 环境标识。可覆盖命令行 `-env` 参数 |

---

### 3.2 `logging` — 日志组件

**这是最基础的组件，几乎所有其他组件都依赖它。建议始终启用。**

```yaml
logging:
  enabled: true
  level: "DEBUG"                    # 日志级别
  format: "json"                    # 输出格式
  output: "file"                    # 输出目标
  file_config:                      # output=file 时必须配置
    dir: "./logs"
    filename: ""                    # 留空则自动使用 app_name
  rotate_config:                    # 日志轮转
    enabled: true
    rotate_interval: 24h
    max_age: 168h                   # 7 天
    cleanup_enabled: true
```

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `enabled` | bool | — | 是否启用日志组件 |
| `level` | string | `"INFO"` | 日志级别: `DEBUG` / `INFO` / `WARN` / `ERROR` |
| `format` | string | `"json"` | 输出格式: `json`（结构化）/ `console`（人类可读） |
| `output` | string | `"stdout"` | 输出目标: `stdout` / `stderr` / `file` |
| `file_config.dir` | string | — | 日志文件目录（output=file 时必填） |
| `file_config.filename` | string | `app_name` | 日志文件名前缀。留空自动用 `app_info.app_name` |
| `rotate_config.enabled` | bool | false | 是否启用日志文件轮转 |
| `rotate_config.rotate_interval` | duration | — | 轮转间隔。 `≥24h` 则按天生成 `app.log.YYYYMMDD` |
| `rotate_config.max_age` | duration | — | 日志保留时间（配合 cleanup_enabled 使用） |
| `rotate_config.cleanup_enabled` | bool | false | 是否自动删除过期日志 |

**使用方式（业务代码中）**：

```go
import "github.com/grand-thief-cash/chaos/app/infra/go/application/components/logging"

logging.Info(ctx, "server started")
logging.Infof(ctx, "listening on %s", addr)
logging.Warnf(ctx, "slow query: %s dur=%v", query, dur)
logging.Error(ctx, "connection failed", zap.Error(err))
```

### 3.3 `http_server` — HTTP 服务器

```yaml
http_server:
  enabled: true
  address: ":8080"                 # 监听地址
  read_timeout: 30s                # 读取超时
  write_timeout: 30s               # 写入超时
  idle_timeout: 60s                # 空闲连接超时
  graceful_timeout: 10s            # 优雅关闭超时
  enable_health: true              # 启用 /health 端点
  enable_pprof: false              # 启用 /debug/pprof 端点
```

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `address` | string | `":8080"` | 监听地址（格式: `:端口` 或 `IP:端口`） |
| `read_timeout` | duration | `30s` | 服务器读取请求（header + body）的最大时间。防止 slowloris 攻击 |
| `write_timeout` | duration | `30s` | 服务器写响应的最大时间。防止 handler 因客户端慢读而挂起 |
| `idle_timeout` | duration | `60s` | HTTP/1.1 keep-alive 空闲连接保持时间。释放不活跃客户端的资源 |
| `graceful_timeout` | duration | `10s` | 关闭时等待 in-flight 请求完成的最大时间 |
| `enable_health` | bool | false | 启用 `GET /health` 端点（返回所有组件健康状态） |
| `enable_pprof` | bool | false | 启用 `GET /debug/pprof/*` 端点（**生产环境建议关闭**） |

**注册路由**：

```go
app := application.GetApp()

// 获取 HTTP Server 组件并注册路由
comp, _ := app.GetComponent("http_server")
httpServer := comp.(*http_server.HTTPServerComponent)
httpServer.RegisterRoutes(func(mux *http.ServeMux) {
    mux.HandleFunc("/api/v1/hello", helloHandler)
})
```

### 3.4 `http_clients` — HTTP 客户端（支持多个命名客户端）

```yaml
http_clients:
  enabled: true
  default: "phoenixa"               # 默认客户端名
  clients:
    phoenixa:
      base_url: "http://localhost:18085"
      timeout: 10s
      max_idle_conns: 200
      max_idle_conns_per_host: 100
      idle_conn_timeout: 90s
      default_headers:
        X-Source: "atlas"
      retry:
        enabled: true
        max_attempts: 3
        initial_backoff: 100ms
        max_backoff: 2s
        backoff_multiplier: 2.0
    external_api:
      base_url: "https://api.example.com"
      timeout: 30s
```

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `default` | string | `"default"` | 默认客户端名称（用于 `GetDefault()` 调用） |
| `clients.<name>.base_url` | string | — | 基础 URL（所有请求自动拼接） |
| `clients.<name>.timeout` | duration | `10s` | 单次请求超时 |
| `clients.<name>.max_idle_conns` | int | `200` | 全局空闲连接池上限 |
| `clients.<name>.max_idle_conns_per_host` | int | `100` | 每个 host 的空闲连接上限 |
| `clients.<name>.idle_conn_timeout` | duration | `90s` | 空闲连接存活时间 |
| `clients.<name>.default_headers` | map | — | 每次请求自动附加的 Header |
| `clients.<name>.retry.enabled` | bool | false | 是否启用重试 |
| `clients.<name>.retry.max_attempts` | int | `3` | 最大重试次数 |
| `clients.<name>.retry.initial_backoff` | duration | `100ms` | 首次重试延迟 |
| `clients.<name>.retry.max_backoff` | duration | `2s` | 最大重试延迟 |
| `clients.<name>.retry.backoff_multiplier` | float | `2.0` | 延迟指数增长倍数 |

### 3.5 `grpc_server` — gRPC 服务器

```yaml

  enabled: true
  address: ":50051"
  max_recv_msg_size: 10485760      # 10MB
  max_send_msg_size: 10485760
  graceful_timeout: 10s
```

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|

| `enable_reflection` | bool | false | 启用 Server Reflection（支持 grpcurl 调试） |
| `enable_health` | bool | false | 启用 gRPC Health Check 服务 |

### 3.6 `grpc_clients` — gRPC 客户端（支持多个命名连接）

```yaml
grpc_clients:
  enabled: true
  default_timeout: 5s
  enable_health_check: true
  health_check_interval: 30s
  clients:
    user_service:
      name: "user-service"
      host: "localhost"
      port: 50051
      secure: false
      timeout: 5s
      connect_on_start: true
      max_receive_message_length: 10485760
      max_send_message_length: 10485760
      compression: "gzip"
      retry_policy:
        max_retries: 3
        initial_delay: 100ms
        max_delay: 2s
        multiplier: 2.0
      keepalive_options:
        time: 30s
        timeout: 10s
        permit_without_stream: true
```

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `default_timeout` | duration | `5s` | 所有客户端的默认调用超时 |
| `enable_health_check` | bool | false | 是否启用连接健康检查 |
| `health_check_interval` | duration | `30s` | 健康检查间隔 |
| `clients.<name>.host` | string | — | 目标服务地址 |
| `clients.<name>.port` | int | — | 目标服务端口 |
| `clients.<name>.secure` | bool | false | 是否使用 TLS |
| `clients.<name>.credentials_path` | string | — | TLS 证书路径（secure=true 时使用） |
| `clients.<name>.timeout` | duration | `default_timeout` | 该客户端的调用超时（覆盖默认值） |
| `clients.<name>.connect_on_start` | bool | false | 启动时立即建立连接（否则懒连接） |
| `clients.<name>.compression` | string | — | 压缩方式: `gzip` / 留空不压缩 |
| `clients.<name>.retry_policy.max_retries` | int | `3` | 最大重试次数 |
| `clients.<name>.retry_policy.initial_delay` | duration | `100ms` | 首次重试延迟 |
| `clients.<name>.retry_policy.max_delay` | duration | `2s` | 最大重试延迟 |
| `clients.<name>.retry_policy.multiplier` | float | `2.0` | 延迟增长倍数 |
| `clients.<name>.keepalive_options.time` | duration | `30s` | 发送 keepalive ping 的间隔 |
| `clients.<name>.keepalive_options.timeout` | duration | `10s` | 等待 keepalive ack 的超时 |
| `clients.<name>.keepalive_options.permit_without_stream` | bool | false | 无活跃 stream 时是否仍发送 keepalive |

### 3.7 `mysql_gorm` — MySQL 数据库（GORM，支持多数据源）

```yaml
mysql_gorm:
  enabled: true
  log_level: "info"                # GORM 日志级别
  slow_threshold: 200ms            # 慢查询阈值
  data_sources:
    security:                      # 数据源名称（业务代码中用这个名字获取连接）
      host: 127.0.0.1
      port: 3306
      user: root
      password: "123456"
      database: security
      params:                      # 额外 DSN 参数
        charset: utf8mb4
      max_open_conns: 50
      max_idle_conns: 10
      conn_max_life: 60m
      conn_max_idle: 10m
      ping_on_start: true
      skip_default_tx: true
      prepare_stmt: false
      migrate_enabled: true
      migrate_base: ./migrations   # → auto-resolves to ./migrations/mysql/security/
```

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `log_level` | string | `"warn"` | GORM 内部日志级别: `silent` / `error` / `warn` / `info` |
| `slow_threshold` | duration | `200ms` | 慢查询日志阈值 |
| `data_sources.<name>.dsn` | string | — | 完整 DSN（配了此项则忽略 host/port 等字段） |
| `data_sources.<name>.host` | string | — | 数据库地址 |
| `data_sources.<name>.port` | int | `3306` | 端口 |
| `data_sources.<name>.user` | string | — | 用户名 |
| `data_sources.<name>.password` | string | — | 密码 |
| `data_sources.<name>.database` | string | — | 数据库名 |
| `data_sources.<name>.params` | map | — | 额外连接参数（如 `charset`, `loc` 等） |
| `data_sources.<name>.max_open_conns` | int | `0`(不限) | 最大打开连接数 |
| `data_sources.<name>.max_idle_conns` | int | `0` | 最大空闲连接数 |
| `data_sources.<name>.conn_max_life` | duration | `0`(不限) | 连接最大生命周期。建议设置（如 `60m`），避免因数据库端关连接导致报错 |
| `data_sources.<name>.conn_max_idle` | duration | `0` | 空闲连接最大存活时间 |
| `data_sources.<name>.ping_on_start` | bool | false | 启动时是否 ping 数据库验证连通性 |
| `data_sources.<name>.skip_default_tx` | bool | false | GORM 默认每次操作包裹事务。设 true 提升读多写少场景性能 |
| `data_sources.<name>.prepare_stmt` | bool | false | 启用预编译语句缓存（提高复用 SQL 性能） |
| `data_sources.<name>.migrate_enabled` | bool | false | 是否在启动时执行 SQL 迁移 |
| `data_sources.<name>.migrate_base` | string | — | 迁移文件根目录。自动解析为 `{base}/mysql/{name}/` |

**使用方式**：

```go
comp, _ := app.GetComponent("mysql_gorm")
mysqlComp := comp.(*mysqlgorm.GormComponent)
db, err := mysqlComp.GetDB("security")  // 获取命名数据源的 *gorm.DB
```

### 3.8 `postgres_gorm` — PostgreSQL 数据库（GORM，支持多数据源 + 多 Schema + 扩展）

```yaml
postgres_gorm:
  enabled: true
  log_level: "info"
  slow_threshold: 200ms
  data_sources:
    security:                       # 数据源名称
      host: 192.168.1.100
      port: 5432
      user: chaos_app
      password: "secret"
      database: chaos_db
      schema: public                # PostgreSQL search_path
      params:
        sslmode: disable
      max_open_conns: 50
      max_idle_conns: 10
      conn_max_life: 60m
      conn_max_idle: 10m
      ping_on_start: true
      skip_default_tx: true
      enable_timescale: true        # 自动创建 TimescaleDB 扩展
      enable_pgvector: false
      migrate_enabled: true
      migrate_base: ./migrations   # → auto-resolves to ./migrations/postgresql/security/
    kg:                             # 第二个数据源（同一 DB 不同 schema）
      host: 192.168.1.100
      port: 5432
      user: chaos_app
      password: "secret"
      database: chaos_db
      schema: kg                    # 自动 SET search_path TO kg
      max_open_conns: 20
      max_idle_conns: 5
      conn_max_life: 60m
      ping_on_start: true
      enable_pgvector: true         # 自动创建 PGVector 扩展
      migrate_enabled: true
      migrate_base: ./migrations   # → auto-resolves to ./migrations/postgresql/kg/
```

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `data_sources.<name>.schema` | string | — | **PostgreSQL 专属**。设置后自动执行 `SET search_path TO <schema>` + 创建 schema |
| `data_sources.<name>.enable_timescale` | bool | false | 启动时自动 `CREATE EXTENSION IF NOT EXISTS timescaledb` |
| `data_sources.<name>.timescale_schema` | string | — | TimescaleDB 扩展安装到哪个 schema（通常留空） |
| `data_sources.<name>.enable_pgvector` | bool | false | 启动时自动 `CREATE EXTENSION IF NOT EXISTS vector` |

> 其余参数（host/port/user/password/database/max_open_conns/...）与 `mysql_gorm` 相同，参见上文。

**TimescaleDB Hypertable 操作**：

```go
comp, _ := app.GetComponent("postgres_gorm")
pgComp := comp.(*postgresgorm.PostgresGormComponent)

// 将普通表转为 TimescaleDB hypertable
err := pgComp.EnsureHypertable(ctx, "security", "bars_stock_zh_a_1min", "trade_date", "1 week")
```

### 3.9 `redis` — Redis 客户端

```yaml
redis:
```

| 参数 | 类型 | 默认值 | 说明 |

| `read_timeout` | duration | `3s` | 读取超时 |

### 3.10 `prometheus` — Prometheus 监控端点

```yaml
prometheus:
```

| 参数 | 类型 | 默认值 | 说明 |

| `collect_process` | bool | true | 是否采集进程级指标（CPU / FD 数 / 启动时间） |

### 3.11 `neo4j` — Neo4j 图数据库客户端

```yaml
neo4j:

  max_connection_pool_size: 50       # 连接池大小

```

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `uri` | string | `"bolt://localhost:7687"` | Neo4j Bolt 协议地址。格式: `bolt://host:port` 或 `neo4j://host:port`（后者支持路由） |
| `username` | string | — | 认证用户名 |
| `password` | string | — | 认证密码 |
| `database` | string | `"neo4j"` | 目标数据库名。Neo4j Community 版只有一个数据库 `neo4j` |
| `max_connection_pool_size` | int | `50` | 驱动内部连接池的最大连接数 |
| `encrypted` | bool | false | 是否启用 TLS 加密连接 |

**特性**：
- 启动时自动验证连接可达性（`VerifyConnectivity`）
- 内置 `HealthCheck`：调用 `VerifyConnectivity` 检测连接状态
- 自动处理 Neo4j Node / Relationship / Path 等类型到 JSON-serializable `map[string]any` 的转换

**使用方式**：

```go
comp, _ := app.GetComponent("neo4j")
neo4jComp := comp.(*neo4j.Neo4jComponent)

// 只读查询
rows, err := neo4jComp.RunCypher(ctx, "MATCH (c:Company {name: $name}) RETURN c", map[string]any{"name": "宁德时代"})
// rows = []map[string]any{ {"c": {"_labels": ["Company"], "name": "宁德时代", ...}} }

// 写入查询
affected, err := neo4jComp.RunCypherWrite(ctx, `
    MERGE (c:Company {normalized_name: $name})
    ON CREATE SET c.name = $name, c.created_at = datetime()
    ON MATCH SET c.updated_at = datetime()
`, map[string]any{"name": "宁德时代"})
// affected = 受影响的 nodes/relationships/properties 总数

// 获取底层驱动（高级用法）
driver := neo4jComp.Driver()
```

**返回值序列化规则**：

| Neo4j 类型 | 转换后 Go 类型 |
|-----------|---------------|
| `Node` | `map[string]any`，含 `_labels` ([]string) 和 `_id` (string) |
| `Relationship` | `map[string]any`，含 `_type`、`_id`、`_startId`、`_endId` |
| `Path` | `map[string]any`，含 `nodes` 和 `relationships` 列表 |
| 原始类型 | 直接保留（string, int64, float64, bool, []any） |

### 3.12 `telemetry` — OpenTelemetry 分布式追踪与指标

```yaml
telemetry:
  enabled: true                    # 是否启用
  service_name: "phoenixA"         # 服务名（留空则取 app_name）
  exporter: "stdout"               # 导出器类型
  sample_ratio: 1.0                # 采样率 (0~1)
  stdout_pretty: false             # stdout 导出器是否美化输出
  stdout_file: "./logs/telemetry.log"  # 写入文件而非终端
  # 文件轮转（stdout_file 非空时生效）
  file_max_size_mb: 100            # 单文件最大 MB
  file_max_age_days: 7             # 保留天数
  file_max_backups: 5              # 最多保留旧文件数
  # OTLP 导出器配置（exporter=otlp 时使用）
  otlp:
    endpoint: "localhost:4317"
    insecure: true
    timeout: "5s"
```

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `enabled` | bool | — | 是否启用。`false` = 创建 no-op provider（安全调用 API 但不输出任何数据） |
| `service_name` | string | `app_name` | 在 trace/metrics 中标识的服务名 |
| `exporter` | string | `"none"` | 导出器: `none`（不输出）/ `stdout`（终端/文件）/ `otlp`（远程 Collector） |
| `sample_ratio` | float | `1.0` | 采样率。`0` = 不采样，`1.0` = 全量采样。生产环境建议 `0.1` |
| `stdout_pretty` | bool | false | stdout 导出器输出是否美化 JSON |
| `stdout_file` | string | — | 非空则将 telemetry 输出写入此文件（支持轮转） |
| `file_max_size_mb` | int | `100` | 单文件最大大小 (MB)，超过则轮转 |
| `file_max_age_days` | int | `7` | 旧日志文件最大保留天数 |
| `file_max_backups` | int | `5` | 最多保留几个旧日志文件 |
| `otlp.endpoint` | string | — | OTLP Collector 地址 |
| `otlp.insecure` | bool | false | 是否使用不安全连接（开发环境） |
| `otlp.timeout` | string | `"5s"` | OTLP 导出超时 |

**Telemetry 控制逻辑**：

```
telemetry 配置节不存在       → 组件不创建（完全跳过）
telemetry.enabled: false    → 组件创建但以 no-op 模式运行（API 安全调用，无输出）
telemetry.exporter: "none"  → 同上（provider 初始化但不导出）
telemetry.exporter: "stdout" + stdout_file 为空  → 输出到终端
telemetry.exporter: "stdout" + stdout_file 非空  → 输出到文件（带轮转）
telemetry.exporter: "otlp"  → 发送到 OTLP Collector
```

## 四、Migration（数据库迁移）机制

### 4.1 概述

Go Infra 内置了版本追踪的 SQL 迁移系统，支持 MySQL 和 PostgreSQL。

**核心特性**：
- 每个数据源维护一张 `_migrations` 追踪表
- 只执行尚未应用的迁移文件（已执行的自动跳过）
- 迁移文件按文件名字典序排序执行
- PostgreSQL 正确处理 `$$` dollar-quoting（函数/触发器）
- 每个迁移文件在事务中执行（原子性）

### 4.2 目录结构规范

迁移文件按 **数据库类型 → 数据源名称** 分层组织：

```
migrations/                         ← 根目录（配置中的 migrate_base）
├── postgresql/                     ← 数据库类型
│   ├── security/                   ← 数据源名称（对应 data_sources 下的 key）
│   │   ├── 0001_init.sql
│   │   └── 0002_add_timescale.sql
│   └── kg/                         ← 另一个数据源
│       ├── 0001_kg_init.sql
│       └── 0002_kg_add_embeddings.sql
├── mysql/                          ← MySQL 类型
│   └── legacy/                     ← MySQL 数据源名称
│       └── 0001_init.sql
└── clickhouse/                     ← 预留：未来扩展
    └── analytics/
        └── 0001_init.sql
```

**路径自动解析**：

配置中只需指定 `migrate_base`（根目录），组件自动拼出完整路径：

```
{migrate_base}/{dialect}/{datasource_name}/
```

示例：
| migrate_base | 组件 | 数据源 | 解析为 |
|-------------|------|--------|--------|
| `./migrations` | postgres_gorm | `security` | `./migrations/postgresql/security/` |
| `./migrations` | postgres_gorm | `kg` | `./migrations/postgresql/kg/` |
| `./migrations` | mysql_gorm | `legacy` | `./migrations/mysql/legacy/` |

### 4.3 迁移文件命名规范

```
0001_init.sql                       # 编号_描述.sql
0002_add_indexes.sql
0003_create_function.sql
```

**命名规则**：
- 文件名必须以 `.sql` 结尾
- 建议使用 4 位数字前缀（`0001_`, `0002_`）保证排序
- 文件名中的描述用 `_` 分隔
- 文件名作为唯一标识存入 `_migrations` 表（**不要改已执行的文件名！**）

### 4.4 配置

```yaml
postgres_gorm:
  data_sources:
    security:
      migrate_enabled: true
      migrate_base: ./migrations   # 根目录 → 自动解析为 ./migrations/postgresql/security/
      schema: public               # _migrations 表也在这个 schema 下
    kg:
      migrate_enabled: true
      migrate_base: ./migrations   # 同一根目录 → 自动解析为 ./migrations/postgresql/kg/
      schema: kg                   # → kg._migrations 表

mysql_gorm:
  data_sources:
    legacy:
      migrate_enabled: true
      migrate_base: ./migrations   # → 自动解析为 ./migrations/mysql/legacy/
```

> 所有数据源可以共享同一个 `migrate_base`，组件根据自身类型和数据源名称自动定位到正确的子目录。

### 4.5 追踪表结构

框架自动创建：

```sql
-- PostgreSQL（如有 schema，创建为 <schema>._migrations）
CREATE TABLE IF NOT EXISTS kg._migrations (
    id         BIGSERIAL PRIMARY KEY,
    filename   VARCHAR(512) NOT NULL UNIQUE,
    applied_at TIMESTAMP NOT NULL DEFAULT NOW(),
    checksum   VARCHAR(64) DEFAULT ''
);

-- MySQL
CREATE TABLE IF NOT EXISTS _migrations (
    id         INT AUTO_INCREMENT PRIMARY KEY,
    filename   VARCHAR(512) NOT NULL UNIQUE,
    applied_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    checksum   VARCHAR(64) DEFAULT ''
);
```

### 4.6 编写迁移文件的注意事项

1. **使用 `IF NOT EXISTS`**：让迁移文件幂等，即使手动执行也安全
2. **不要修改已应用的文件**：框架按文件名追踪，不会重新执行
3. **一个文件只做一件事**：便于回溯
4. **PostgreSQL 函数/触发器**：使用 `$$` 包裹（框架会正确处理）

```sql
-- 0003_create_function.sql
CREATE OR REPLACE FUNCTION update_modified_column()
RETURNS TRIGGER AS $$
BEGIN
   NEW.updated_at = NOW();
   RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

## 五、组件间依赖和启动顺序

框架自动按依赖关系排序组件的启动顺序：

```
logging                    ← 最先启动（其他组件的日志依赖）
    ↓
telemetry                  ← 依赖 logging
prometheus                 ← 依赖 logging
mysql_gorm                 ← 依赖 logging
postgres_gorm              ← 依赖 logging
redis                      ← 依赖 logging
neo4j                      ← 依赖 logging
grpc_clients               ← 依赖 logging
    ↓
http_server                ← 可能依赖 prometheus
grpc_server                ← 可能依赖 telemetry
```

**关闭顺序**：与启动顺序相反（先停 HTTP/gRPC 服务器，最后停日志）。

## 九、常见问题

### Q1: 组件不启动？
**A**: 检查配置文件中该组件的 `enabled: true` 是否设置。如果整个配置节未出现，组件也不会创建。

### Q2: 迁移文件重复执行？
**A**: v0.17.0 之前的版本没有追踪表，每次启动都会重新执行所有 `.sql` 文件。升级到 v0.17.0+ 后，首次启动会创建 `_migrations` 表并记录所有文件，后续启动自动跳过已执行的文件。

### Q3: 迁移文件怎么组织（migrate_base vs 旧 migrate_dir）？
**A**: v0.17.0 起使用 `migrate_base` 替代旧的 `migrate_dir`。只需指定根目录，组件自动按 `{base}/{db_type}/{datasource}/` 解析。不同数据源共享同一个 `migrate_base` 即可。

### Q3: PostgreSQL 的 `$$` 函数定义报语法错误？
**A**: v0.17.0+ 已支持 `$$` dollar-quoting。确保使用最新版本的 Go Infra。

### Q4: 多个数据源连同一个数据库不同 schema？
**A**: 在 `data_sources` 中配置多个名字不同的数据源，指向同一个 `database`，设置不同的 `schema` 即可。

### Q5: Telemetry 日志文件太大？
**A**: v0.17.0+ 支持文件轮转,配置 `file_max_size_mb`、`file_max_age_days`、`file_max_backups` 即可。
