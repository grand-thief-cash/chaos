# Backlog: 派生数据存储需求

## 背景

当某些数据需要从多个数据源组合计算得到（如市值 = 收盘价 × 总股本），当前策略是**实时计算**（每次使用时从多个数据源读取并计算）。

这种方式的缺点：
- 增加查询延迟（需要多次 HTTP 请求）
- 增加数据源负载
- 无法复用计算结果
- 数据一致性风险（计算时刻的不同数据源状态）

## 期望方案

将派生数据（如市值）存储在 PhoenixA 中，避免重复计算。但这需要：
- 依赖多个下载任务都完成
- 需要触发计算时机
- 需要设计存储表结构

## 当前派生数据需求

### 1. 市值数据 (Market Cap)

**计算公式**：
```
market_cap = close_price × TOT_SHARE
```

**数据来源**：
- `close_price`: 来自 `bars_daily` 表 (via `GET /api/v2/bars/stock/zh_a`)
- `TOT_SHARE`: 来自 `financial_statement.data_json` (balance_sheet)

**因子用途**：
- PE (市盈率) = market_cap / 净利润_TTM
- PB (市净率) = market_cap / 净资产
- PS (市销率) = market_cap / 营业收入_TTM
- PCF (市现率) = market_cap / 经营现金流_TTM

**暂定方案**：
- 在 Factor Engine 中实时计算（每次计算因子时调用）
- 记录到本 backlog，等待更多类似需求后统一设计

### 2. 未来可能的派生数据

- 市盈率 PE (TTM / 静态)
- 市净率 PB
- 股本变化追踪（用于复权计算）
- 技术指标 (MACD, KDJ, RSI 等常用指标)

## 统一设计的考虑因素

当收集到更多派生数据需求后，统一设计时应考虑：

1. **依赖管理**
   - 派生数据依赖的原始数据表
   - 需要跟踪原始数据更新时间
   - 计算触发时机（原始数据更新后？定时任务？）

2. **计算节点**
   - 是否需要在 Artemis 中增加计算节点？
   - 还是在 PhoenixA 中使用 SQL 计算？
   - 或者使用新的计算引擎服务？

3. **增量更新策略**
   - 按日更新（市值）还是按事件触发？
   - 历史数据回溯如何处理？

4. **数据一致性**
   - 派生数据的时间点如何与原始数据对齐？
   - 交易日 vs 报告期 vs 公告日期的混用场景

5. **存储设计**
   - 独立表还是扩展现有表？
   - 是否需要版本管理（因子版本对应不同计算逻辑）？
